# Welo · Absenteeism Analytics Mockup

A presentation dashboard for Welo's absenteeism intelligence, **wired to the real
absenteeism model output**. Every headline number, risk band, fatigue segment,
cohort breakdown and individual prediction on the page is generated by the model
in [`model/`](model/), not hardcoded.

Open [`index.html`](index.html) in a browser. It hydrates itself from
`model/data/outputs/dashboard_feed.js`. If the feed is missing it still renders
with the designed fallback numbers, so it never breaks for a screenshot.

## What the dashboard shows

The dashboard is organised around **cohorts**, not sites. The model's signal is
operational and clinical (fatigue/burnout, smoking + alcohol load, metabolic
load, workload), so the workforce is grouped by the levers Welo can actually act
on rather than by geography.

- **Portfolio** - model-level headline: covered lives, predicted absent days
  (90d), cost exposure, addressable saving, risk distribution, fatigue/burnout
  segment, driver concentration, and a cohort summary table.
- **Cohorts** - group the covered workforce by **operational load**, **fatigue
  band**, **age**, **tenure** or **lifestyle risk**. Each cohort card shows its
  high/critical concentration, days per head, mean fatigue, cost exposure and
  addressable saving. Click a cohort to drill in.
- **Individuals** - the model scores every covered life. High and critical
  employees are listed first. Filter by risk band or cohort, sort, search, and
  click any row to open a profile drawer with the full prediction: risk-band
  probabilities, fatigue score, the drivers behind the prediction (SHAP where
  available, otherwise the model features sitting above the cohort norm), the
  employee's profile, and their cohort memberships.
- **HR & Ops** - the operational view for the HR team: predicted absence
  rate, shift cover-gap days and overtime backfill cost, absence frequency
  (Bradford-style), return-to-work caseload and leave hygiene, broken down by
  operational cohort, plus an HR action playbook. Cost and spell assumptions
  are benchmark-anchored and documented in the feed's `hr_ops.assumptions`.
- **Outcomes & ROI** - projected addressable saving, seasonality and an
  intervention scenario.

> The individual records are synthetic mining-cohort rows, not real people. Every
> synthetic row is tagged `synthetic_mining_v1__glencore_sa_mining` so it can be
> swapped for real client data without touching the dashboard.

### Why cohorts instead of sites

Three reasons, open to challenge:

1. **The model has no site signal.** It predicts from health, fatigue and
   workload features. Cohorts defined on those features line up with where the
   absence actually concentrates, e.g. the *smoker + drinker* cohort runs a
   33% high/critical share vs 3% for *low lifestyle risk*; *high-intensity ops*
   runs 22% vs 6% for *light duty*. Site-level cuts wash that out.
2. **Cohorts map to interventions.** A high-fatigue or high-overtime cohort has
   an obvious rostering/clinical response; "Site 2" does not.
3. **Site data may not exist on day one.** Cohorts are derivable from the feed
   the model already produces, so the view works before any HR/location join.

When real site/department fields arrive they become just another cohort lens;
the Cohorts screen already supports multiple dimensions.

## The model (`model/`)

`model/` is the full absenteeism stack: a config-driven training pipeline
(`welo_pipeline`), a FastAPI inference service (`welo_inference`), trained
artifacts (`models/`), and the cached dashboard feed (`data/outputs/`). See
[`model/MODEL_README.md`](model/MODEL_README.md) for the pipeline, the inference
service, and the Cloud Run deployment.

Model performance (cross-validated, from `model/reports/model_metrics.json`):
regression CV R² ≈ 0.40 / MAE ≈ 6.0 hours; classification CV ROC-AUC (macro OvR)
≈ 0.84.

## Regenerating the dashboard feed

The base feed (headline, risk distribution, fatigue, intervention queue with
SHAP reasons, global importance, metrics) comes from the training pipeline:

```bash
cd model
python -m welo_pipeline --config configs/demo.yaml
```

The dashboard additionally needs **cohort aggregates** and **per-individual
records**. Those are added by `model/build_cohort_feed.py`, which reproduces the
deterministic synthetic cohort (seed 7, n=5800, validated to match the
predictions 1:1 on actual absence hours), joins it to `predictions.csv`, derives
the cohort dimensions, and writes the enriched `dashboard_feed.json` / `.js`:

```bash
cd model
pip install numpy pandas pyyaml
python build_cohort_feed.py
```

Then reopen `index.html`.
